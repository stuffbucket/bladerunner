# Instance Floppies — adversarial review (must-resolve before Phase 2)

Produced by the design workflow critic, verified against lxc/incus/v6@v6.23.0.

## Verdict

Strong, evidence-grounded PRD + scope; the metaphor, the import/export pivot, the data-safety surface, and the phase structure are sound and the codebase anchors are real (I verified the SDK methods, attach/eject/provisioning anchors, and the useHosted=false default). But it is NOT yet build-ready: several load-bearing claims about the Incus backup API contradict how that API actually works, the "optimized-storage" checkpoint path collides with the "fall back to dir" story, and the most-repeated guarantee ("never delete the VM-pool copy until export is confirmed") is mechanically impossible as written because export materializes the backup ON the pool first. Fix the API-contract blockers below and tighten the clobber/multi-machine edge cases before Phase 2 starts.

## Blockers (fix before building Phase 2)

### CONTRADICTS THE SDK: export is a two-step materialize-then-download flow, not a stream. CreateInstanceBackup (internal/incus, verified at incus_instances.go:2882) writes a FULL backup tarball onto the guest pool's backup storage; only then does GetInstanceBackupFile (io.WriteSeeker, interfaces.go:464) stream it to the DMG. So a checkpoint/eject of an N-GB floppy transiently needs ~N GB FREE on the btrfs pool, on top of the live instance. The PRD/scope never budget this. With several floppies (courier+fresco) checkpointing, the loopback btrfs pool (default 20GiB in spike #2) can ENOSPC mid-checkpoint, and an ENOSPC during the server-side backup create is a failure mode the anti-clobber matrix doesn't cover.

**Fix:** Add a pool free-space precondition + sizing model: pool capacity must be >= sum(live floppies) + max(single floppy backup) + headroom; make the first-boot btrfs pool size a function of expected floppy count, not a fixed 20GiB. Treat 'backup-create ENOSPC' as a first-class anti-clobber case (abort checkpoint, keep last good DMG, loud warn). Delete the server-side backup record (DeleteInstanceBackup) in a defer so failed checkpoints don't leak pool space.

### THE HEADLINE SAFETY INVARIANT IS MIS-STATED. 'Never delete the VM-pool copy until the export is confirmed' is presented as the load-bearing guarantee, but the live instance is NEVER deleted during checkpoint/eject in the first place — it keeps running in the pool. What actually must be protected is the EXISTING good DMG payload (the .prev), not 'the VM-pool copy'. As written the invariant is either a no-op or implies deleting the running instance, which would be a bug.

**Fix:** Restate as: 'never overwrite/discard the last verified DMG payload (.prev) until the new export is fsync'd+verified+renamed; on eject, do not Detach (and do not DeleteInstance unless the user asked) until WriteBack returns success.' Make the eject path explicit about whether the pool instance is deleted on eject at all (PRD J3 says 'detach' but never says the instance is removed from the pool — clarify, because leaving it means re-insert collides on instance name).

### OPTIMIZED-STORAGE CHECKPOINT vs DIR-FALLBACK CONTRADICTION. Phase 2 hardcodes OptimizedStorage=true (btrfs send) for cheap checkpoints, but Phase 0 explicitly allows a loud fallback to a DIR pool when btrfs is unavailable. An optimized/btrfs-send backup is driver-specific: it is produced by btrfs and is NOT restorable into a dir pool, and OptimizedStorage on a dir pool is meaningless/unsupported. So a floppy checkpointed optimized on machine A cannot be inserted on a machine B whose pool fell back to dir — a silent cross-machine import failure the PRD's 'importing the same DMG on two machines' case never considers.

**Fix:** Make optimized a function of the actual pool driver at runtime (query storage pool driver; only set OptimizedStorage when driver==btrfs). Stamp the DMG manifest with backup_format=optimized|portable and the source driver; on insert, refuse/auto-convert an optimized payload when the target pool is dir, with a clear error. Default eject (the portable savefile) should arguably be NON-optimized for portability, using optimized only for fast intra-machine checkpoints.

### NEWER-INCUS / EXTENSION-VERSION GAP IS UNADDRESSED. CreateInstanceFromBackup requires API extensions container_backup_override_pool and backup_override_name (verified in incus_instances.go:516-548) to import under a chosen name/pool — exactly what insert needs. A DMG whose backup was produced by a NEWER Incus (the PRD's 'export made by a newer Incus' edge case) can carry a backup-format/header version the target guest's older Incus rejects. There is no version field in the floppy Manifest and no preflight extension check.

**Fix:** Record incus_version + backup_format_version in the Manifest at export time; on insert, preflight HasExtension() for the required extensions and compare backup-format version, failing fast with a clear 'this floppy needs a newer machine' message instead of an opaque import error. Add this to the smoke matrix.

### IMPORT IS A SINGLE OPAQUE TARBALL — the 'split the DMG into metadata + payload' model needs a defined reassembly contract. InstanceBackupArgs.BackupFile is one io.Reader (interfaces.go:599). The PRD/scope describe storing (a) a small read-mostly metadata file and (b) the rootfs payload separately in the DMG, but never specify how those recombine into the single native backup tarball Incus requires, nor whether the 'metadata' is the floppy's own manifest or Incus's backup index.yaml. Ambiguity here is where insert silently breaks.

**Fix:** Define precisely: the DMG holds (1) floppy-owned sidecars (Stamp, floppy Manifest) and (2) the UNMODIFIED native incus backup tarball as the payload. ImportInstance feeds the payload tarball verbatim as io.Reader; the 'small metadata Incus needs' is INSIDE that tarball (backup index), not a separately reassembled file. Drop any language implying bladerunner reconstructs the tarball from parts.

## Gaps to resolve

- Clobber matrix is incomplete: it covers DMG overwritten/deleted/read-only/force-detached, but NOT (a) instance-name collision on insert when an instance of that name already runs in the pool (two copies of the same template, or a stale leftover from a crashed session), (b) ENOSPC on the pool during checkpoint/eject, (c) a checkpoint failing partway leaving a server-side backup record orphaned, (d) crash DURING WriteBack between fsync-temp and rename (recovery must detect and resume/discard the temp). Each needs a defined degrade-to-warn behavior.
- Re-insert / pool-residue lifecycle is undefined. After eject 'detaches', is the instance deleted from the pool? If not, re-inserting the same DMG (or inserting a save-as child) collides on instance name; if yes, the 'never delete pool copy' language conflicts. Importing the SAME DMG twice (PRD edge case) into one machine MUST be handled (auto-suffix name? refuse? clone?) and the registry key is name+stamp, so two inserts of one stamp need an explicit rule.
- Multi-floppy resource limits are asserted but unbounded. No cap on concurrent floppies, no per-floppy CPU/mem profile, and the checkpoint goroutine ticks ALL floppies every 5 min with a per-floppy lock — but nothing staggers them, so N large floppies can fire N concurrent btrfs sends + N pool-materializations at the same instant, contending for pool IO and space. Spike #3's 4s/GB non-pausing number was single-instance; it is not shown to hold under concurrent checkpoints. Add staggering/jitter and a concurrency cap, and re-measure the multi-floppy case in smoke.
- Name-collision surface beyond instance name: btrfs/optimized exports and VirtioFS share tags (Phase 5) both need globally-unique tags; the registry keys on name+stamp but the Incus instance namespace, the snapshot names (CreateInstanceSnapshot uses a snapshot name — needs a collision-free scheme, e.g. checkpoint-<ts>), and future share tags are separate namespaces each needing a collision rule. Snapshot accumulation/pruning is also unspecified (every checkpoint snapshots; nothing prunes — pool fills).
- Phase independence is overstated for Phase 0 and Phase 1. Phase 1 ('insert, it runs') is NOT genuinely shippable as user value without Phase 2: insert with no checkpoint/eject means the only way to persist is... nothing, and there's no eject, so an inserted floppy can only be torn down by stopping the machine, losing all work. Phase 1 is an internal milestone, not a shippable increment. Be honest that v1 user value starts at Phase 2. (Phase 0 IS genuinely independent — that claim holds.)
- darwin/GOOS=linux obligation has a specific trap the scope underplays: WriteBack's fsync-of-the-DMG-mount semantics differ by host. On macOS, fsync() does NOT flush to stable storage (needs F_FULLFSYNC); a plain Sync() can return before data hits the platter, defeating the atomicity guarantee on power loss. The data-safety code lives in pure-Go (Linux CI) but the REAL host is darwin — so the very platform where the guarantee matters is the one not exercised by the unit tests. Need a darwin-gated F_FULLFSYNC path and a smoke-level (not unit) durability check.
- No verification that the pre-baked image's Incus version actually carries the required extensions (container_backup_override_pool etc.). The btrfs-init snippet is asserted to be applied in 4 places, but the smoke test only checks 'btrfs default pool exists' — it does not assert the guest Incus exposes the import-override extensions, which is the actual load-bearing capability. Add an extension-presence assertion to smoke-floppy Phase 0.
- Eject timing/no-pause claim needs re-grounding: spike #3 proved EXPORT didn't pause a RUNNING container. But eject does SetInstanceState(Stop) FIRST, then exports a STOPPED instance — different path, and the 'export while guest still live' wording in the risk section contradicts the 'graceful stop then final export' flow in J3. Clarify whether the final eject export is of a stopped or running instance; the ejectWaitMargin sizing depends on it.
- Verb 'seal'/'save-as'/'checkpoint' collision check is partial. The review correctly clears disk/boot/eject/cartridge, but 'save-as' visually neighbors the reserved VZ 'save', and 'floppy new --template' overlaps 'disk new' semantics enough to confuse users. Worth a help-text disambiguation and a test asserting 'br save' / 'br eject' (bare) still route to VZ machine-state, not floppy.

## Strengths (validated)

- The core architectural pivot is correctly grounded: spike #1 (VirtioFS idmap collapse) genuinely kills live-rootfs-on-DMG, and the import/export-into-pool model is the right response. I verified CreateInstanceFromBackup/CreateInstanceBackup/GetInstanceBackupFile all exist in lxc/incus/v6@v6.23.0 with the cited shapes, so the SDK foundation is real, not aspirational.
- Codebase anchoring is unusually honest and mostly accurate: attachArgs hardcodes a writable mount (cartridge.go:165, confirmed — AttachReadOnly is genuinely new), assets.go does rename-without-fsync/.prev (confirmed ~308, so the 'greenfield write-back' framing is right), useHosted defaults false (config.go, confirmed), and ejectWaitMargin=15s / saveCommandTimeout=10min are real (eject.go:89, listener.go:26). An implementer can trust the map.
- The build-tag discipline is correctly specified: copying the cartridge_darwin.go/cartridge_other.go split with !darwin ErrUnsupported stubs, and the explicit 'keep exported API referenced on Linux to dodge unused-code lint' is exactly the right guardrail for this repo's CI.
- Decisions are made, not deferred-by-default: verb noun, checkpoint cadence, container-only v1, bake-progs-init-pool-on-first-boot, and explicit DEFER of DiskArbitration + live share with sound rationale. The 'do not flip useHosted until -no-agent naming is reconciled' callout is a genuine, correctly-identified 404 trap.
- The data-safety design instinct is correct even where the invariant wording is off: temp-in-DMG -> fsync -> verify(size+sha256) -> atomic rename -> one .prev, plus identity-stamp conflict side-files, is the right shape and the fault-injection-as-the-gate framing for Phase 2 is appropriately paranoid.
- Phase 0's '4 divergent provisioning paths, miss one => silent dir pool' risk and the 'no || true swallow' rule are real, well-targeted, and exactly the kind of drift this repo is prone to (the existing incus admin init --auto || true at cloudinit.go:372 is precisely the line to kill).
